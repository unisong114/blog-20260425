# GitHub Pages 블로그 만들기 4. 필수 파일 구성하기

테마를 연결했다고 해서 바로 블로그가 완성되는 것은 아니다.  
블로그처럼 보이게 하려면 홈 화면, 소개 페이지, 글 저장 위치 같은 기본 뼈대를 먼저 갖춰야 한다.

이번 글에서는 Jekyll 블로그에서 자주 쓰는 필수 파일들을 정리해본다.

## 가장 기본이 되는 파일들

프로젝트를 처음 세팅할 때는 보통 아래 파일부터 만든다.

- `index.md`
- `about.md`
- `_posts/`
- `_config.yml`
- `.gitignore`
- `README.md`

이 파일들이 각각 어떤 역할을 하는지 알고 있으면 이후 확장이 훨씬 쉬워진다.

## index.md는 홈 화면이다

`index.md`는 사이트 첫 화면이 된다.  
Jekyll은 이 파일을 정적 페이지로 렌더링해 메인 화면으로 보여준다.

예시는 아래처럼 시작할 수 있다.

```md
---
title: Home
nav_order: 1
description: GitHub Pages blog home
---

# Unisong Blog

이 사이트는 GitHub Pages 위에서 운영되는 블로그입니다.
```

여기에 블로그 소개, 최근 글 목록, 시작 안내 같은 내용을 넣으면 첫 인상이 훨씬 좋아진다.

## about.md는 블로그 소개 페이지다

블로그를 어떤 목적으로 운영하는지 설명하는 페이지가 하나쯤 있으면 좋다.

예시는 아주 단순해도 충분하다.

```md
---
title: About
nav_order: 2
---

# About

이 블로그에서 다루는 주제와 운영 목적을 소개합니다.
```

처음부터 길게 쓰지 않아도 된다.  
짧아도 방향이 보이는 소개가 있으면 사이트의 인상이 훨씬 또렷해진다.

## _posts 폴더는 블로그 글이 쌓이는 곳이다

Jekyll에서 일반적인 블로그 글은 `_posts` 폴더 아래에 저장한다.

파일 이름은 아래 규칙을 따른다.

```text
YYYY-MM-DD-title.md
```

예를 들면:

```text
2026-04-25-first-post.md
```

이 규칙을 지켜야 Jekyll이 날짜 기반 포스트로 인식한다.

## 포스트의 front matter

각 글의 상단에는 YAML front matter를 넣는다.

예시는 아래와 같다.

```md
---
title: "첫 글 제목"
date: 2026-04-25 17:00:00 +0900
categories: [github-pages, jekyll]
tags: [blog, setup]
---
```

이 정보는 제목, 날짜, 분류, 태그처럼 글의 메타데이터 역할을 한다.

## .gitignore도 초반에 꼭 넣자

Jekyll 프로젝트는 빌드 과정에서 로컬 전용 파일이 생길 수 있다.  
그런 파일까지 저장소에 올라가면 관리가 지저분해진다.

예를 들어 아래 항목들은 보통 제외한다.

```text
_site/
.jekyll-cache/
.jekyll-metadata
.bundle/
vendor/
Gemfile.lock
```

작은 파일처럼 보여도 초반에 정리해두면 저장소 관리가 훨씬 편하다.

## README.md는 미래의 나를 위한 안내서다

처음에는 사소해 보이지만 `README.md`는 꽤 중요하다.  
이 저장소가 무엇인지, 어떻게 실행하는지, 어디로 배포되는지 적어두면 나중에 다시 볼 때 큰 도움이 된다.

보통 아래 정도를 적어두면 충분하다.

- 이 저장소가 블로그 저장소라는 점
- 로컬 실행 방법
- 배포 방식
- 실제 배포 주소

## 파일 구조를 갖추는 이유

파일 몇 개를 만드는 일은 단순해 보이지만, 사실은 블로그의 운영 방식을 정리하는 과정이다.

- 홈에서 무엇을 보여줄지
- 소개 페이지를 둘지
- 글은 어떤 규칙으로 쌓을지
- 저장소는 어떻게 관리할지

이 기준이 초반에 정해져야 글이 늘어나도 구조가 무너지지 않는다.

## 마무리

Jekyll 블로그는 처음부터 거창한 기능이 없어도 괜찮다.  
오히려 핵심 파일 몇 개를 안정적으로 갖추는 편이 훨씬 오래 간다.

다음 글에서는 GitHub Pages 배포 설정과 GitHub Actions 워크플로를 연결해, 저장소의 변경이 실제 사이트로 이어지게 만드는 과정을 정리해보겠다.
